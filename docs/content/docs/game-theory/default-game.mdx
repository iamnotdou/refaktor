# Default Oyunu

Lidl ödemeyebilir. Bu **stratejik** mi yoksa **dış şok** mu? Game theory bunu ayırır.

## İki Tür Default

### Strategic Default
Buyer ödeyebilir ama **ödememeyi seçer** — daha karlı çünkü:
- Yasal recourse zayıf (cross-border)
- Reputation kaybı geri kazanımdan az

### Liquidity Default
Buyer **ödeyemiyor** — fiilen iflas, kriz, vs.

## Strategic Default Hesabı

Buyer'ın game tree:
```
Buyer (Lidl)
├── Öde (50.000€)
│   ├── KOBİ memnun → tekrar iş
│   ├── Reputation korumalı
│   └── Net cost: 50.000€
└── Ödeme (0€)
    ├── KOBİ ile iş bitti (-future revenue)
    ├── Yasal süreç (-legal cost, ~30k EUR)
    ├── Reputation kaybı (-other suppliers korkar)
    └── Net "kazanç": 50.000 - 30.000 - reputation_cost
```

Reputation kayıp **uzun vadeli** → 50.000€'luk savings'ten büyük olabilir → öder.

## FaturaChain'in Caydırıcılığı

FaturaChain on-chain reputation getirir:
- Buyer'ın default geçmişi public, immutable
- Bir kez default → tüm KOBİ'ler ondan kaçar (sadece o KOBİ değil)
- Multi-supplier kaybı

Bu **multilateral reputation**, Lidl gibi büyük alıcılar için çok yüksek maliyet. Strategic default neredeyse imkansız hale gelir.

## Liquidity Default

Buyer iflas etti — yapacak bir şey yok. Burada:
- Insolvency proceedings yıllar sürer
- Pro-rata recovery (alacaklılar listesinde)
- Genelde %20-40 recovery

Bu yüzden:
- A rating buyer (Lidl, Bosch) → liquidity default %0.1
- B rating (orta firma) → %1
- C rating → %5
- D rating → %15+ (kabul edilmez)

## KOBİ'nin Strategic Default'u

KOBİ de "default benzeri" davranabilir:
- Sahte fatura çıkarır → para alır → kaybolur
- GİB doğrulama bunu engellemeli (mock'tan gerçeğe Phase 2)

### Korunma Mekanizmaları

| Mekanizma | Etkisi |
|---|---|
| GİB e-Fatura cross-check | Fatura gerçek mi? |
| MFKS sorgulama | Aynı fatura başka yerde finanse edilmiş mi? |
| KOBİ KYC | Sahte kimlik koymasın |
| Overcollateralization | KOBİ sahte fatura çıkarsa %15'i kaybeder |
| Repeat business | Reputation oluşumu |

## Compound Game — Multi-period

Buyer her dönem 10 fatura için ödeme kararı verir. **Tit-for-tat** stratejisi:
- Dönem 1: KOBİ tedarik et + fatura kes → Buyer öde
- Dönem 2: Tekrar
- ... dönem N: Toplam ilişki değeri >> tek seferde "kazanç"
- Yani **cooperation equilibrium** — herkes öder

Folk theorem: yeterince düşük discount rate'le sonsuz tekrarlanan oyun cooperation Nash equilibrium'a sahip.

## Bilgi Açığa Çıkması (Default Cascade)

Bir buyer default ettiğinde:
1. O buyer'ın diğer faturaları için bid'ler hızla iptal olur
2. Aynı sektördeki buyer'lar için risk premium artar
3. Aynı ülkedeki buyer'lar etkilenir (sistemsel)

Bu **information cascade** krize sebep olabilir. Korunma:
- Anti-correlation portföy çeşitlendirme
- Sigorta havuzu çapraz dengeleme
- Circuit breaker (ani default'ta pazarı durdur)

## Hackathon'da Default Demo

Demo'da bir invoice'i kasıtlı default ettirip:
1. `escrow.declareDefault(invoiceId)` çağrısı
2. Order book otomatik dondurulur
3. `InsurancePool.payClaim` token sahiplerine pro-rata ödeme
4. Sigorta havuzu bakiyesi azalır
5. Default rate dashboard güncellenir

Bu jüriye **sistem default'ı zarafetle yönetiyor** mesajını verir. Çoğu Web3 RWA projesi bu adımı atlıyor.

## Default Sonrası Pazar Dinamiği

Default → buyer rating C/D'ye düşer → yeni faturaları yüksek iskontolu satılır (%85'den %75'e) → yatırımcılar bu rating'e güvenmiyor → buyer pazardan elenir.

Bu **otomatik karantina** mekanizması — kötü buyer'ı sistem kendi kendine eler.

## Şüpheli Davranış (Phase 2)

Buyer aniden:
- Çok büyük bir fatura listeniyor (anormal volume)
- Birkaç KOBİ aynı anda aynı buyer'a fatura kesti

Bu sinyaller AI fraud detection ile yakalanır → admin manuel inceleme.

## Net Mesaj

Default tamamen önlenemez ama **fiyatlanır**. Açık pazarda buyer rating'i yatırımcı bilir, premium ödemez, KOBİ daha kötü fiyat alır. Bu **doğru fiyatlama** sigortayı sürdürülebilir kılar.
