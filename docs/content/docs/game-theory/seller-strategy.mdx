# KOBİ Stratejisi

KOBİ iki kararı verir:
1. **Token sayısı** ($n$) — kaç parçaya bölünecek
2. **Ask price** ($p_S$) — token başına kaç para

## Token Sayısı Kararı

### Az Token (örn. 50)
- Token başına büyük (50.000 / 50 = 1.000€ nominal)
- Sadece büyük yatırımcılar girer → likidite düşük
- Spread büyük → KOBİ kayıp eder

### Çok Token (örn. 5000)
- Token başına küçük (10€ nominal)
- Küçük yatırımcılar girebilir → fraksiyonel demokratik
- Order book spam riski (çok küçük emirler gas yer)

### Optimal: 500 Token
- Token başına 100€ → minimum giriş 100€
- Yeterli granularite, spam yok
- "Standart denominasyon" → karşılaştırma kolay

`★ Insight ─────────────────────────────────────`
**100€ standart denominasyon** Polytrade ve diğer Web3 platformlarda da var. Yatırımcılar bu birime alışkın → bilişsel yük az. Hackathon demoda 500 token / 100€ nominal ile gidip "endüstri standartı" demek jüriye güven verir.
`─────────────────────────────────────────────────`

## Ask Price Kararı — Üç Yaklaşım

### Yaklaşım 1: Maliyet Bazlı (en zayıf)
KOBİ kendi maliyet+ kar marjına göre ask koyar.
- "Faturanın %95'ini istiyorum" → 95€
- Pazar dinamiğini görmez

### Yaklaşım 2: Pazar Bazlı (en iyi)
Aynı rating + benzer vade faturaların **son trade fiyatlarını** bak:
```
Last 30 trade'de ortalama: 94.8€
Mevcut order book best bid: 95.2€
→ Optimal ask: 95.0-95.3€ aralığı
```

### Yaklaşım 3: Aciliyet Bazlı (panik)
Ahmet'in çok hızlı nakte ihtiyacı varsa best bid'e ask koyar:
```
Best bid: 94€ (Mehmet)
KOBİ: 94€ ask → anlık fill, ama 1k EUR kaybetti
```

## Acil Nakit vs Maximum Getiri Trade-off

| KOBİ tipi | Strateji | Beklenen ortalama |
|---|---|---|
| Nakit aciliyeti var | Best bid'i hit et | %92 nominal |
| Normal akış | İlk denemede 95€, beklemezse düşür | %94.5 nominal |
| Yatırım perspektifi | Yüksek koy, beklerken faiz var | %96 nominal (riskli) |

## Dinamik Ask Güncelleme (Optimal)

Demo MVP'sinde manuel, Phase 2'de **otomatik**:

```python
def update_ask(invoice, current_book):
    days_held = invoice.listed_days
    target_fill_rate = 0.95   # %95 satılsın istiyoruz
    current_fill = invoice.tokens_sold / invoice.total_tokens
    
    if days_held > 7 and current_fill < 0.5:
        # Yavaş satıyor, ask düşür
        return current_ask * 0.99
    elif current_fill > 0.8:
        # Hızlı bitiyor, ask yükselt
        return current_ask * 1.005
    else:
        return current_ask
```

## Token Tutma Stratejisi (Kısmi Satış)

KOBİ 500 tokenin hepsini satmak zorunda değil:
```
Sat 300 token (anlık 28.5k nakit)
Tut 200 token (kendi yatırımı, 100€'a kavuşacak — 5k kazanç)
```

Bu KOBİ'ye **kendi faturasının yatırımcısı** olma fırsatı verir. Kendine güveniyorsa profit.

## Game Tree Özeti

```
KOBİ
├── Ask çok yüksek (97€)
│   ├── Yatırımcı bid yetersiz → satış yok
│   └── Bekleyip sonra düşür → günlük opportunity cost
├── Ask doğru (95€)
│   ├── Hızlı eşleşme → optimal ✓
│   └── Kısmi fill → kalan ile ne yapsın?
├── Ask agresif (93€)
│   ├── Anında full fill → garantili nakit
│   └── 2k kazanç bıraktı
```

## Behavioral Bias'lar

KOBİ'lerde gözlenen biaslar:
- **Anchoring:** Faturanın nominal değerini "yapışkan referans" olarak görür → ask çok yüksek koyar
- **Loss aversion:** %5 iskonto ödemekten çok rahatsız olur ama 30 gün beklemeyi tolere eder (mantıksız)
- **Overconfidence:** "Lidl kesin öder" diye düşünüp risk premium istemez

Platform UX'i bu biasları azaltmalı:
- "Last 30 trade'de %95 nominal'den satıldı" → anchoring zayıflatma
- "Bu fiyatla bekleme süresi: ~3 gün" → bekleme maliyeti görselleştirme
