# Auditor / Compliance

Sistemi **dışarıdan izleyen ve denetleyen** rol. Yazma yetkisi yok, sadece okuma. BDDK/SPK uyumu, vergi otoritesi (GİB), dış denetim firması bu rolü kullanır.

## Profil

- **Tipik örnek:** GİB e-Fatura denetim, BDDK fintech şubesi, 4 büyük denetim firması
- **Erişim:** Read-only API + dashboard
- **Hedef:** Şüpheli işlem tespiti, vergi raporlaması, lisans uyumluluğu

## Görevleri

1. Belirli bir KOBİ'nin tüm faturalarını listeleme
2. Belirli bir alıcıya kesilen faturaların toplam hacmi
3. Default olan fatura listesi ve etkilenen yatırımcılar
4. Anti-money-laundering (AML) — büyük transferleri izleme
5. Vergi raporu üretme (yatırımcının yıllık getirisi)

## Sistemden Beklediği

- **Read-only API** (audit_role JWT)
- **Tüm transaction logu** (on-chain zaten public ama metadata + KYC bilgisi off-chain)
- **Time-stamped immutable** kayıtlar
- **Filtre + export** — CSV / PDF
- **GİB e-Fatura cross-check** — sistemdeki fatura GİB'de var mı

## On-chain vs Off-chain

| Veri | Yer | Auditor erişim |
|---|---|---|
| Token sahipliği | On-chain (public) | Direkt blockchain explorer |
| Ödeme akışı | On-chain (USDC transferi) | Direkt |
| KOBİ KYC bilgisi | Off-chain (encrypted DB) | API (ihtiyaç anında) |
| Alıcı bilgisi | On-chain hash + off-chain detay | API |
| GİB e-Fatura referansı | Off-chain | API + GİB API cross-check |

## Compliance Sinyalleri

Auditor şunları otomatik tespit etmek ister:
- **Wash trading** — aynı kullanıcının kendi tokenına bid + ask koyup volüm yaratması
- **Layering** — birden fazla cüzdandan koordineli alım
- **Suspicious volume** — bir KOBİ aniden 10× fatura listelerse
- **Front-running** — admin'in özel bilgi kullanarak alım yapması
